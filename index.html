This document describes a minimal set of topics to be covered by a basic introduction to PackML. Instructors are free to include any additional information, including vendor-specific information, deemed necessary.

Content order

1. Introduction

This section is intended to give students an overview of the basic concepts and development history of PackML.

After completing the section, students should have an understanding of what motivated the development of PackML and some of its benefits.

1.1 PackML goals

Instructors: Present an overview of the goals and benefits of PackML. Some ideas are listed below.

PackML, or Packaging Machine Language, is an effort by OMAC to standardize the interface between a packaging machine and its peers and supervisors. Primary objectives include bringing common "look and feel" and operational consistency to all machines that make up a packaging line.

PackML provides:

  • Overall equipment effectiveness or OEE data
  • Root cause analysis or RCA data
  • Recipe management
  • Standard defined machine states and operational flow

What does it mean for machines to have a common "look and feel"?

Give an example of operational consistency.

1.2 Development timeline

Instructors: It may be helpful to review the developemnt history of PackML. The chart below was adapted from https://en.wikipedia.org/wiki/PackML. Further information is available there.

1.3 Current standard

Instructors: Stress the fact that PackML is defined by ANSI/ISA-TR88.00.02-2015. Clarify how this document may be otained.

1.4 Data interface

Instructors: Introduce the idea that one of PackML's central roles is to function as a data interface between the machine and its supervisors and peers.

Give examples of data that might be passed from a machine to its supervisor.

Give examples of data that might be passed between machine peers.

1.5 PackTags

Instructors: Introduce the term "PackTags" and relate this term to the previous section: the set of PackTags form the complete data interface.

Why is it critical to have naming conventions for the data included in the PackML data interface?

1.6 PackML state machine

Instructors: Preview the PackML state machine. Stress the idea that in order to have a cohesive data interface, a common understanding of machine state is required. Later sections will cover the indivual machine states in detail.

Clarify the state machine workflow, especially the meaning of the boxes connected to the Stop and Aborted commands.

In non-technical terms, give examples of common machine conditions or states (e.g. machine is running, error has occurred, etc.).

2. Terms and key concepts

This section introduces key terms and concepts.

After completing the section, students - in addition to understanding the terms themselves - should have a clearer understanding of the scope of the course.

2.1 PackML

PackML: a communication interface for discrete manufacturing (e.g. packaging machinery) described by ISA-TR88.00.02-2015; sometimes used to describe additional methods of standardizing machine “look and feel” or PLC coding design patterns.

2.2 State or machine state

State or machine state: the current status or condition of the system in consideration.

Choose a machine and describe its states. For example, what are the states of an automobile?

2.3 State model

State model: an abstract model or machine that can be in exactly one of a finite number of states at any given time. The model may change or transition from one state to another in response to some input.

What happens if you push the turnstile barrier without inserting a coin? How is this feature represented in the diagram above?

2.4 ISA-88 physical model

ISA-88 physical model: a hierarchy describing the physical assets of a commercial enterprise. See ANSI/ISA-S88.01-1995. In this model, a Process Cell comprises one or more Units responsible for producing a specific commodity.

2.5 Mode or machine mode

Mode (general): a way or manner in which something occurs or is experienced, expressed or done.

Unit/machine control mode: a way or manner in which a manufacturing machine is used.

Mode manager: software that governs the transition from one machine mode to another.

2.6 PackTags

PackTags: a set of naming conventions for software variables associated to data elements of the unit/machine.

2.7 Alarm handler

Alarm handler: software responsible for logging alarms or events occurring on the unit/machine. Functionality for sorting or filtering the alarm history may also be present. Some prefer the more general term event handler.

3. PackML states

This chapter covers the PackML state model, describing in detail each of the states, as well as the overall workflow.

After completing the chapter, students should have a clear understanding of the PackML state model.

3.1 EXECUTE state

Instructors: Introduce the EXECUTE state. Stress the idea that if the machine is running and producing product, then from a PackML perspective it is in EXECUTE state.

Reinforce the notion that the PackML state is associated to specific machine behaviors. It is not up to the programmer to decide whether a particular PackML state is used. Rather, the machine's operational behaviour determines what the PackML state must be.

Throughout this chapter it is recommended to concentrate on the machine behaviour associated to each state in Production mode, as these are in a sense the most well defined. PackML modes will be discussed in the next chapter, but instructors may want to briefly introduce the concept of mode here.

If the unit/machine is actively carrying out the behavior or activity defined by the selected mode, then the machine is in EXECUTE state. If the unit/machine is in PRODUCTION mode, for example, this means that the machine is producing product.

Table 1 (p. 17): Once the machine is processing materials it is in the EXECUTE state. Different machine modes will result in specific types of EXECUTE activities. For example, if the machine is in the “Production” mode, the [sic] EXECUTE will result in products being produced, while in “Clean Out” mode the EXECUTE state refers to the action of cleaning the machine.

Respond to this description of the PackML state machine: The PackML state tells you whether the machine is running or not and if not, why not.

3.2 HELD state

Refer to HOLDING for when this state is used. In this state the machine shall not produce product. It will either stop running or continue to dry cycle. A transition to the UNHOLDING state will occur when INTERNAL machine conditions change or an UNHOLD command is initiated by an operator.

The unit/machine is paused, waiting for internal process conditions to clear. In this state, the unit/machine shall not produce product, though it may, if required, dry-cycle. A transition to UNHOLDING will occur once internal machine conditions have cleared or if the UNHOLD command is initiated by an operator.

3.3 SUSPENDED state

Refer to SUSPENDING for when this state is used. In this state the machine shall not produce product. It will either stop running or continue to cycle without producing until external process conditions return to normal, at which time, the SUSPENDED state will transition to the UNSUSPENDING state, typically without any operator intervention.

The unit/machine is paused, waiting for external process conditions to clear. In this sate, the unit/machine shall not produce product, but may, if required, dry-cycle. Once external conditions have returned to normal, the unit/machine will transition to UNSUSPENDING, typically without operator intervention.

3.4 COMPLETE state

The machine has finished the COMPLETING state and is now waiting for a RESET command before transitioning to the RESETTING state.

COMPLETE indicates the process associated with the active mode has come to its defined end. The unit/machine will wait in this state until a RESET command is issued (in which case it will transition to RESETTING), or until the process is STOPPED or ABORTED.

3.5 IDLE state

This is the state which indicates that RESETTING is complete. The machine will maintain the conditions which were achieved during the RESETTING state, and perform operations required when the machine is in IDLE.

The unit/machine is in an error-free state, waiting to start. The unit/machine transitions automatically to IDLE after all steps necessary for RESETTING have been completed. All conditions achieved during RESETTING are maintained. A START command will transition the unit/machine from IDLE to STARTING.

3.6 STOPPED state

The machine is powered and stationary after completing the STOPPING state. All communications with other systems are functioning ( if applicable). A RESET command will cause an exit from STOPPED to the RESETTING state.

The unit/machine is powered and stationary after either STOPPING or CLEARING. All communication with supervisory, user interface, peer or other systems is active and functioning. A RESET command will cause the unit/machine to transition from STOPPED to the RESETTING state.

3.7 ABORTED state

The machine maintains status information relevant to the ABORT condition. The machine can only exit the ABORTED state after an explicit CLEAR command, subsequently to manual intervention to correct and reset the detected machine faults.

The unit/machine remains in its ABORTED state until a CLEAR command is explicitly issued. Note that operators may be required to take manual, corrective action in this state. The unit/machine must maintain all status or fault information related to the cause of the current ABORT condition.

3.8 HELD vs. SUSPENDED

The unit/machine will transition from EXECUTE to HOLDING if conditions internal to the unit/machine require a pause in production. Such conditions would include low levels on materials required for production or other minor issues requiring operator service, for example. The decision to HOLD may be made automatically by the unit/machine itself or by an operator.

The unit/machine will transition from EXECUTE to SUSPENDING if conditions external to the unit/machine require a pause in production. Such conditions include faults to upstream or downstream equipment. The decision to SUSPEND may be made be a supervisory system monitoring the production line conditions or by unit/machine sensors detecting downstream blockages or upstream product scarcity. (In the former case, the unit/machine is 'blocked”; in the latter case, the unit/machine is “starved”.)

3.9 STOPPED vs. ABORTED

The unit/machine transitions to STOPPING in response to a STOP command. In this state the unit/machine performs all steps required to bring itself to a controlled stop.

The unit/machine transitions to ABORTING in response to an ABORT command or an internal machine fault. In this state the unit/machine performs all steps required to bring itself to a rapid, safe stop.

6. Implementation strategies

The purpose of this chapter is to give students practical advice on how PackML may be implemented in code.

Vendors or technology provides are asked to provide details of their own libraries or toolsets, including a review of best practices.

Two specific subjects should be included. The first, alarm handling, should include a description of how this handler interacts with the PackML state machinde. The second, code structure, should include both gateway (adding PackML to an existing code base) and full integration (PackML from the ground up) approaches.

6.1 Alarm handling

6.2 Code structure

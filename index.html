PackML Implementation Guide

Part X: Teaching PackML Basics

Summary

This document provides a framework for a basic course on PackML. It is meant as a resource for instructors and aims to help harmonize courses on this topic currently being offered by various technology providers.

The course outline covers the major points found in TR88.00.02-2015, including the PackML state machine, PackML modes and PackTags. Additional sections describing PackML's overarching goals and development history should serve to motivate students. Throughout, vendors are encouraged to add content specific to their own technologies.

Supplemental content

This document describes a minimal set of topics to be covered by a basic introduction to PackML. Instructors are free to include any additional information, including vendor-specific information, deemed necessary.

Content order

Topics are given in what the authors feel is a natural order, but the subject matter may be rearranged where required, especially with respect to any additional vendor-specific material.

1. Introduction

Chapter 1 is intended to give students an overview of the basic concepts and development history of PackML.

After completing the chapter, students should have an understanding of what motivated the development of PackML and its main benefits.

1.1 PackML goals

Instructors: Present an overview of the goals and benefits of PackML. Some key ideas are listed below.

PackML, or Packaging Machine Language, is an effort by OMAC to standardize the interface between a packaging machine and its peers and supervisors. Primary objectives include bringing common "look and feel" and operational consistency to all machines that make up a packaging line.

PackML provides:

  • Overall equipment effectiveness or OEE data
  • Root cause analysis or RCA data
  • Recipe management
  • Standard defined machine states and operational flow

What does it mean for machines to have a common "look and feel"?

Give an example of operational consistency.

PackML has found application beyond packaging to discrete manufacturing generally. What is discrete manufacturing?

1.2 Development timeline

Instructors: Briefly review the development history of PackML. The chart below was adapted from https://en.wikipedia.org/wiki/PackML. Further information is available there.

Stress that PackML is both well-established and currently active.

1.3 Current standard

Instructors: Make the point that PackML is defined by ANSI/ISA-TR88.00.02-2015. Clarify how this document may be obtained.

1.4 Data interface

Instructors: Introduce the idea that one of PackML's central roles is to function as a data interface between the machine and its supervisors and peers. Clarify the terms peer and supervisor.

Give examples of data that might be passed from a machine to its supervisor.

Give examples of data that might be passed between machine peers.

1.5 PackTags

Instructors: Introduce the term "PackTags" and relate this term to the previous section: the set of PackTags form the complete data interface. Preview the overall scope of the data set.

Why is it critical to have naming conventions for the data included in the PackML data interface?

What other conventions regarding the data interface are required?

1.6 PackML state machine

Instructors: Preview the PackML state machine. Stress the idea that in order to have a cohesive data interface, a common understanding of machine state is required. Later sections will cover the individual machine states in detail.

Clarify the state machine workflow, especially the meaning of the boxes connected to the Stop and Abort commands.

In non-technical terms, give examples of common machine conditions or states (e.g. machine is running, error has occurred, etc.).

How does having a common understanding of machine state facilitate the creation of a standardized calculation for overall equipment effectiveness (OEE)?

2. Terms and key concepts

Chapter 2 introduces key terms and concepts.

After completing the chapter, students - in addition to understanding the terms themselves - should have a clearer understanding of the scope of the course and, by extension, the scope of PackML itself.

2.1 PackML

PackML: a communication interface for discrete manufacturing (e.g. packaging machinery) described by ISA-TR88.00.02-2015; sometimes used to describe additional methods of standardizing machine “look and feel” or PLC coding design patterns.

2.2 State or machine state

State or machine state: the current status or condition of the system in consideration.

Choose a machine and describe its states. For example, what are some states of an automobile?

2.3 State model

State model (finite): an abstract model or machine that can be in exactly one of a finite number of states at any given time. The model may change or transition from one state to another in response to some input.

What happens if you push the turnstile barrier without inserting a coin? How is this feature represented in the diagram above?

2.4 ISA-88 physical model

ISA-88 physical model: a hierarchy describing the physical assets of a commercial enterprise. See ANSI/ISA-S88.01-1995. In this model, a Process Cell comprises one or more Units responsible for producing a specific commodity.

One of the units shown in the figure above is the Filler. Give an example of one of its (possible) equipment modules. Given an example of one of its control modules.

In the ISA-88 physical model, do the Enteprise and Site levels ever effectively coincide?

2.5 Mode or machine mode

Mode (common meaning): a way or manner in which something occurs or is experienced, expressed or done.

Unit/machine control mode: a way or manner in which a manufacturing machine is used.

Mode manager: software that governs the transition from one machine mode to another.

Give some examples of modes of a packaging machine. In addition to packaging (or producing) product, what other intended uses for a packaging machine are there?

2.6 PackTags

PackTags: a set of naming conventions for software variables associated to data elements of the unit/machine. Where appropriate, the unit of measurement associated with an individual data element is specified.

2.7 Alarm handler

Alarm handler: software responsible for logging alarms or events occurring on the unit/machine. Functionality for sorting or filtering the alarm history may also be present. Some prefer the more general term event handler.

3. PackML states

Chapter 3 covers the PackML state model, describing in detail each of the states, as well as the overall workflow.

After completing the chapter, students should have a clear understanding of the PackML state model.

3.1 EXECUTE state

Instructors: Introduce the EXECUTE state. Stress the idea that if the machine is running and producing product, then from a PackML perspective, it is in EXECUTE state.

Reinforce the notion that PackML state is associated to specific machine behaviors; the machine's operational behavior (and not the programmer's whim) determines what the PackML state must be.

Throughout this chapter it is recommended to concentrate on the machine behavior associated to each state in Production mode, as these are in some sense the most well-defined. PackML modes will be discussed in the next chapter, but instructors may want to briefly review the concept of mode here.

If the unit/machine is actively carrying out the behavior or activity defined by the selected mode, then the machine is in EXECUTE state. If the unit/machine is in PRODUCTION mode, for example, this means that the machine is producing product.

Table 1 (p. 17): Once the machine is processing materials it is in the EXECUTE state. Different machine modes will result in specific types of EXECUTE activities. For example, if the machine is in the “Production” mode, the EXECUTE will result in products being produced, while in “Clean Out” mode the EXECUTE state refers to the action of cleaning the machine.

Respond to this description of the PackML state machine: The PackML state tells you whether the machine is running (i.e. producing product) or not and, if not, why not. (Alternative: The main function of the PackML state machine is to characterize what the machine is doing when it is not producing product.)

3.2 HELD state

Instructors: Introduce the HELD state. Stress that in HELD state, the machine has paused due to internal or local process conditions, typically requiring immediate operator intervention. "Low glue tank" or "Carton tray empty" would be process conditions typically associated with HELD state.

The unit/machine is paused, waiting for internal process conditions to clear. In this state, the unit/machine shall not produce product, though it may, if required, dry-cycle. A transition to UNHOLDING will occur once internal machine conditions have cleared or if the UNHOLD command is initiated by an operator.

Table 1 (p. 17): Refer to HOLDING for when this state is used. In this state the machine shall not produce product. It will either stop running or continue to dry cycle. A transition to the UNHOLDING state will occur when INTERNAL machine conditions change or an UNHOLD command is initiated by an operator.

If a plant manager observes that a particular unit/machine is spending an excessive amount of time in HELD state, what should be his or her response?

3.3 SUSPENDED state

Instructors: Introduce the SUSPENDED state. Stress that in SUSPENDED state, the machine has paused due to external process conditions, typically requiring immediate operator intervention elsewhere (i.e. upstream or downstream) in the process cell. "Downstream blockage" or "Infeed conveyor empty" would be process conditions typically associated with SUSPENDED state.

The unit/machine is paused, waiting for external process conditions to clear. In this sate, the unit/machine shall not produce product, but may, if required, dry-cycle. Once external conditions have returned to normal, the unit/machine will transition to UNSUSPENDING, typically without operator intervention.

Table 1 (p. 17): Refer to SUSPENDING for when this state is used. In this state the machine shall not produce product. It will either stop running or continue to cycle without producing until external process conditions return to normal, at which time, the SUSPENDED state will transition to the UNSUSPENDING state, typically without any operator intervention.

If a plant manager observes that a particular unit/machine is spending an excessive amount of time in SUSPENDED state, what should be his or her response?

Give an example of a machine which might not implement the Suspended state in Production mode.

3.4 COMPLETE state

Instructors: Introduce the COMPLETE state. Stress that in COMPLETE state, the machine behaviour specific to the given mode has come to its logical or assigned conclusion. In production mode, COMPLETE state typically indicates that for a given batch, the required number of products has been produced.

COMPLETE indicates the process associated with the active mode has come to its defined end. The unit/machine will wait in this state until a RESET command is issued (in which case it will transition to RESETTING), or until the process is STOPPED or ABORTED.

Table 1 (p. 17): The machine has finished the COMPLETING state and is now waiting for a RESET command before transitioning to the RESETTING state.

If an operator observes that the unit/machine is at rest in COMPLETE state, what should be his or her response?

3.5 IDLE state

Instructors: Introduce the IDLE state. Stress that in IDLE state, the unit/machine is ready to be started.

The unit/machine is in an error-free state, waiting to start. The unit/machine transitions automatically to IDLE after all steps necessary for RESETTING have been completed. All conditions achieved during RESETTING are maintained. A START command will transition the unit/machine from IDLE to STARTING.

Table 1 (p. 17): This is the state which indicates that RESETTING is complete. The machine will maintain the conditions which were achieved during the RESETTING state, and perform operations required when the machine is in IDLE.

What should an operator do if he or she finds the unit/machine in IDLE state?

3.6 STOPPED state

Instructors: Introduce the STOPPED state. Stress that in STOPPED state, unit/machine operation has stopped, typically due to a decision by an operator or supervisory system, or due to internal machine logic. Point out that the STOPPED state is also reached after CLEARING.

The unit/machine is powered and stationary after either STOPPING or CLEARING. All communication with supervisory, user interface, peer or other systems is active and functioning. A RESET command will cause the unit/machine to transition from STOPPED to the RESETTING state.

Table 1 (p. 17): The machine is powered and stationary after completing the STOPPING state. All communications with other systems are functioning (if applicable). A RESET command will cause an exit from STOPPED to the RESETTING state.

What should an operator do if he or she finds the unit/machine in STOPPED state?

3.7 ABORTED state

Instructors: Introduce the ABORTED state. Associate the ABORTED state with emergency stop, major hardware failures or other severe errors.

The unit/machine remains in its ABORTED state until a CLEAR command is explicitly issued. Note that operators may be required to take manual, corrective action in this state. The unit/machine must maintain all status or fault information related to the cause of the current ABORT condition.

Table 1 (p. 17): The machine maintains status information relevant to the ABORT condition. The machine can only exit the ABORTED state after an explicit CLEAR command, subsequently to manual intervention to correct and reset the detected machine faults.

What should an operator do if he or she finds the unit/machine in ABORTED state?

3.8 HELD vs. SUSPENDED

Instructors: Clarify or review the differences between HELD and SUSPENDED states. The distinction between internal versus external process conditions is key here. Point out that the process of unsuspending is typically done automatically, whereas the process of unholding may (or may not) require operator intervention.

The unit/machine will transition from EXECUTE to HOLDING if conditions internal to the unit/machine require a pause in production. Such conditions would include low levels on materials required for production or other minor issues requiring operator service, for example. The decision to HOLD may be made automatically by the unit/machine itself or by an operator.

The unit/machine will transition from EXECUTE to SUSPENDING if conditions external to the unit/machine require a pause in production. Such conditions include faults to upstream or downstream equipment. The decision to SUSPEND may be made be a supervisory system monitoring the production line conditions or by unit/machine sensors detecting downstream blockages or upstream product scarcity. (In the former case, the unit/machine is 'blocked”; in the latter case, the unit/machine is “starved”.)

3.9 STOPPED vs. ABORTED

Instructors: Clarify or review the differences between STOPPED and ABORTED states. Many view the difference as one of error or event severity, with the ABORTED state being reserved for the most severe errors (e.g. emergency stop, major hardware failure, etc.) One clear distinction provided by the standard is the stopping action required in STOPPING versus ABORTING states: "controlled stop" versus "rapid, safe stop."

The unit/machine transitions to STOPPING in response to a STOP command. In this state the unit/machine performs all steps required to bring itself to a controlled stop.

The unit/machine transitions to ABORTING in response to an ABORT command or an internal machine fault. In this state the unit/machine performs all steps required to bring itself to a rapid, safe stop.

3.10 Acting or transition states

Instructors: Enumerate the acting or transition states: ABORTING, STOPPING, CLEARING, etc. It is suggested to describe them first in a generic or open-ended way: a transition state comprises all actions required to bridge from one wait or stationary state to another. Consider covering one or two transition states in some detail (e.g. Aborting, Starting).

Describe internal machine functions that might occur during Starting state.

Describe internal machine functions that might occur during Aborting state.

3.11 State commands

Instructors: Enumerate the PackML state commands: Abort, Stop, Clear, etc. Point out the obvious correspondence between the acting states and the associated state commands.

Note: TR88.00.02-2021 introduces the Complete command, which initiates a transition from EXECUTE to COMPLETING. It is suggested to introduce this command now, as it is already in wide use and it completes the correspondence between the transition states and state commands.

3.12 State complete

Instructors: In the PackML state diagram, indicate how each the transition from an active state to the subsequent wait state is governed by SC or state complete. Stress that state complete is a logical condition - once state complete, transition to the next state automatically occurs.

On a given machine it is determined that while in STARTING state all servo axes should synchonize with the master axis and all conveyors should begin to run. In general terms, what would the associated state complete condition look like?

In which states may the Abort command be called?

In which states may the Stop command be called?

Which PackML commands are available in the Execute state?

What is the relationship between PackML commands and the acting states?

Each PackML state is assigned a unique fixed number. List these.

Processing on a given machine has paused due to a low level fault on a glue tank. From a PackML perspective, what state is the machine in?

A machine has processed all units required for a given batch and has come to rest. From a PackML perspective, what state is the machine in?

A machine has come to rest due to an emergency stop. From a PackML perspective, what state is the machine in?

Give an example of a machine which might not implement the Suspended state in Production mode.

4. PackML modes

Chapter 4 discusses PackML modes and mode management.

After completing the chapter, students should be able to identify the modes of particular piece of equipment. It should also be clear how the PackML state machine may be redeployed in each mode.

4.1 PRODUCTION mode

Instructors: Introduce PRODUCTION mode. Stress the idea that if the unit/machine is producing product, or attempting to produce product, then from a PackML perspective, it is in PRODUCTION mode.

Reinforce the notion that, while not all PackML states need to be implemented in PRODUCTION mode, the decision to include a particular state or not is determined soley by the machine's operational behavior.

TR88.00.02-2015 stipulates that PRODUCTION mode always be supported in any PackML implementation. Why is this stipulation natural and/or justified?

4.2 MAINTENANCE mode

Instructors: Introduce MAINTENANCE mode. Indicate that MAINTENANCE is an optional mode, used to allow authorized personnel to run the machine independent of other machines in the process cell.

Reiterate that while the machine is actively carrying out the behavior specified by MAINTENANCE mode, the machine is in EXECUTE state.

4.3 MANUAL mode

Instructors: Introduce MANUAL mode. Indicate that MANUAL is an optional mode, used to provide direct control of individual machine modules.

Reiterate that while the machine is actively carrying out the behavior specified by MANUAL mode, the machine is in EXECUTE state.

A capping machine includes functionality to manually test the cap feed system. Would it make sense to include the HELD state in Manual mode in this case?

4.4 Additional modes

Instructors: Provide examples of additional modes that might be present on a packaging machine. In each case, describe the associated PackML state model (or subset thereof) and justify why this set of sub-states was implemented.

4.5 Mode management

Instructors: Introduce the concept of Mode management, the software or logical rules that control the transition from one mode to another.

Stress that when transitioning from one mode to another, PackML stipulates that a simultaneous state change may not occur. A consequence of this rule is that a mode change is only allowed while the machine is in a state common to both modes.

5. PackTags

Chapter 5 focuses on the PackML data interface: PackTags.

After completing the chapter, students should be able to identify the minimal subset of PackTags required for standard compliance. They should understand how the three tag types (Command, Status and Admin) are classified and used. In this basic course, the intention is not to examine each tag in detail, but rather to give a broad overview of the type of information available.

5.1 PackTag types

Instructors: Introduce the three basic PackTag types: Command, Status and Admin.

5.1.1 Command tags

Instructors: Introduce the top-level Command tags. Note that, where appropriate, a specific unit of measure (e.g. primary packages/min) is assumed. Clarify that the Command tags are grouped into a structured data type named PLMc.

Supervisors and peers may write to the Command tags.

5.1.2 Status tags

Instructors: Introduce the top-level Status tags. Note that, where appropriate, a specific unit of measure (e.g. primary packages/min) is assumed. Clarify that the Status tags are grouped into a structured data type named PLMs.

Supervisors and peers may read the Status tags.

5.1.3 Admin tags

Instructors: Introduce the top-level Admin tags. Note that, where appropriate, a specific unit of measure (e.g. primary packages/min) is assumed. Clarify that the Admin tags are grouped into a structured data type named PLMa.

Supervisors and peers may read the Admin tags.

5.2 Base type

Instructors: Structured data type PACKMLV30 comprises the Command, Status and Admin types described previously. Note that TR88.00.02-2015 does not specify the name associated the instance of this type.

Throughout this document we will use UnitName as a placeholder for the actual instance name.
